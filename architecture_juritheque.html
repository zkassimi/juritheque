Architecture Générale

Architecture JAMstack : SPA statique + backend serverless géré + pipeline de données local. Trois couches indépendantes communiquant via API REST.

Couche 1 — Client
Frontend SPA
React 18 · hébergé Hostinger
React 18 Vite 5 React Router v6 Tailwind CSS Lucide React JS/JSX
⇄
Couche 2 — Backend Géré
Supabase
PostgreSQL · Auth · Storage · REST
PostgREST Supabase Auth (JWT) Row Level Security Supabase Storage Supabase Cloud EU
⇄
Couche 3 — Pipeline Local
Python Scripts
Collecte · Extraction · Enrichissement
scraper.py extract.py enrich.py PyMuPDF BeautifulSoup4 httpx
Note d'architecture : Le pipeline Python tourne localement (poste développeur) et pousse les données directement dans Supabase via l'API REST avec la SUPABASE_SERVICE_KEY. Le frontend n'a accès qu'à la clé anonyme avec RLS activé.

06
Sécurité & Row Level Security

RLS activé sur toutes les tables. La clé anon utilisée côté frontend ne peut jamais contourner les politiques.

anon / public
SELECT laws SELECT domains SELECT videos SELECT site_search_index (is_public=true)
user (auth)
SELECT own profile UPDATE own profile SELECT own favorites INSERT own favorites DELETE own favorites
editor (auth)
INSERT laws UPDATE laws INSERT videos + tout user
admin (auth)
Accès complet toutes tables DELETE laws/videos UPDATE tous profils
SQL — Vérification du rôle dans les policies 
-- Pattern utilisé dans toutes les policies editor/admin
CREATE POLICY "Editors update laws"
  ON public.laws FOR UPDATE
  USING (
    exists (
      SELECT 1 FROM public.profiles
      WHERE id = auth.uid()
      AND   role IN ('editor', 'admin')
    )
  );

Contexts & Hooks
🔐
AuthContext
Expose : user, session, profile (role), loading, signIn(), signUp(), signOut().
Sync auto avec onAuthStateChange de Supabase. Profile chargé depuis public.profiles après login.
🌐
LangContext
Expose : lang (fr|ar), isRTL, t(key), toggleLang().
Persisté en localStorage. RTL géré via dir attribute sur <div> root dans App.jsx. Font arabic via Tailwind class.
🔍
useSEO (hook)
Met à jour dynamiquement <title>, meta description, og:*, canonical, hreflang à chaque changement de route ou de contenu. Utilisé dans toutes les pages.
📡
src/lib/api.js
Toutes les fonctions Supabase centralisées : fetchLaws(), fetchLawBySlug(), fetchDomains(), fetchStats(), fetchWatchLaws(), retrieveRelevantSitePages(), etc.

La résolution est en 3 étapes pour garantir la rétrocompatibilité des anciens liens :

JavaScript src/lib/api.js 
async function fetchLawBySlug(slug) {
  // 1. Slug canonique exact → cas normal
  const { data } = await supabase.from('laws')
    .select('*').eq('canonical_slug', slug).maybeSingle()
  if (data) return data

  // 2. Fallback ID numérique → anciens liens /loi/1234
  if (/^\d+$/.test(slug)) { /* ... fetch by id */ }

  // Ex: "dahir-n-1-21-65-dahir-n-1-21-65-du-..." → "dahir-n-1-21-65-du-..."
  const deduped = /* detect & extract non-repeated half */
  if (deduped) { /* try exact, then ILIKE fallback */ }

Authentification & Rôles
🔑 Flux d'authentification
1. Signup → Supabase crée auth.users + déclenche le trigger handle_new_user() → crée automatiquement une ligne dans public.profiles avec role='user'

2. Login → retourne un JWT signé (access token 1h + refresh token 7j)

3. AuthContext écoute onAuthStateChange, charge le profile, expose le rôle dans tout l'app
👤 Rôles utilisateurs
user — lecture publique + favoris personnels

editor — CRUD lois + vidéos, accès back-office limité

admin — accès complet, gestion profils, suppression, analytics

Rôle promu manuellement via Supabase Dashboard ou SQL : UPDATE profiles SET role='editor' WHERE email='...'
🔒 Sécurité clés
VITE_SUPABASE_ANON_KEY — publique, contrainte par RLS

SUPABASE_SERVICE_KEY — bypass RLS, pipeline Python uniquement, jamais côté client

VITE_CLAUDE_API_KEY — Anthropic, appels depuis le navigateur (prévoir migration vers Edge Function)

Problème connu Hostinger : Les modules JS lazy-loadés peuvent avoir des erreurs MIME type (text/html au lieu de application/javascript). Solution appliquée : SEOIntentPage importé en eager dans App.jsx (pas lazy).

Backend — Supabase (géré)

Aucun déploiement manuel. Les migrations SQL sont appliquées via l'éditeur SQL du Dashboard Supabase ou via supabase db push (CLI). Les changements sont instantanément disponibles via l'API REST.

Pipeline — Local uniquement

Le pipeline Python tourne sur le poste développeur et pousse vers Supabase via l'API. Pas de CI/CD pour le pipeline à ce stade.

Sécurité : Clé Claude API côté client
